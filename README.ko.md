# signal-ml-lab

**고전 DSP와 머신러닝으로 생체신호(ECG)를 디노이징하고 정량 비교하는 실험실.**

DSP 알고리즘 개발자가 ML로 넘어가며 만드는 공개 학습 프로젝트. 모든 데이터는
공개 데이터셋([PhysioNet](https://physionet.org/)) 또는 합성 신호이며, 어떤
회사의 독점 데이터·알고리즘도 포함하지 않는다.

## 왜 ECG 디노이징인가
- 생체신호 디노이징은 고전 DSP(필터·웨이블릿)와 학습 기반 방법을
  **같은 지표로 정면 비교**하기 좋은 문제다.
- 노이즈 모델(기저선 변동, 전원선 60Hz, 근전도)이 물리적으로 명확해
  합성 데이터만으로도 정직한 실험이 가능하다.
- 결과가 SNR/RMSE로 딱 떨어져 "그래서 얼마나 나아졌나"를 숫자로 말할 수 있다.

## 지금 되는 것
- 합성 ECG 생성 + 현실적 노이즈 주입 (다운로드 없이 즉시 실행)
- 고전 디노이징: 대역통과 + 60Hz 노치 + (선택)웨이블릿
- **ML 디노이저: 1D DnCNN 잔차 학습 (PyTorch)**
- **고전 vs ML 벤치마크: SNR 스윕 표 + 그림**
- SNR / RMSE 정량 평가, 단위 테스트(pytest)

## 결과 (합성 테스트셋, SNR 스윕)

| input SNR | classical (bp+notch) | ML (DnCNN1D) |
|----------:|---------------------:|-------------:|
|  +5.6 dB  | 5.2 dB               | **20.4 dB**  |
|  −0.4 dB  | 5.0 dB               | **17.3 dB**  |
|  −4.0 dB  | 4.5 dB               | **13.7 dB**  |
|  −6.6 dB  | 3.7 dB               | **9.2 dB**   |
| −10.0 dB  | 2.0 dB               | 2.8 dB       |

- ML은 저~중 노이즈에서 고전 대비 **10~15 dB** 더 좋다.
- **극한 노이즈(−10 dB)에서는 우위가 거의 사라진다** — 정직한 한계.
- 웨이블릿 임계처리는 이 노이즈 모델에선 bandpass+notch 대비 이득이 거의 없다.

![benchmark](assets/03_benchmark_snr.png)

## 실데이터 — sim-to-real 격차 (캡스톤)

합성 이야기가 실신호에서도 버텨야 설득력이 있다. 테스트셋: **실 MIT-BIH ECG**(기준)
+ **실 NSTDB 노이즈**(기저선변동·근전도·전극움직임)를 조절된 SNR로 입힘
(`scripts/08_real_data.py`).

| 방법 | 출력 SNR | 이득 |
|------|---------:|-----:|
| noisy 입력 | 2.83 dB | — |
| classical (bp+notch) | 6.70 dB | +3.87 |
| **ML (합성 학습)** | 4.09 dB | +1.26 |
| **ML (실데이터 학습)** | **9.45 dB** | **+6.62** |

- **합성 학습 모델은 잘 전이되지 않는다** — 실데이터에서 고전 DSP보다도 낮고(4.09 < 6.70),
  R피크를 과하게 솟구치게 만든다(합성 사전지식 강요). 일반화 비용이 합성 변형이 아니라
  **진짜 신호로 확증**됐다.
- **실데이터로 학습하면 ML이 확실히 이긴다**(9.45 dB, 고전 대비 +2.8).
- 교훈: sim-to-real은 실재하는 격차다. 모델은 학습분포의 현실성만큼만 좋다 —
  아키텍처가 분포 불일치를 구해주지 않는다.

![real data](assets/08_real_data.png)

### 해결 — 사전학습 prior로서의 합성 데이터

합성은 쓸모없는 게 아니라 실데이터 학습을 데이터 효율적으로 만드는 **싼 사전학습**
이다 (`scripts/09_finetune_curve.py`).

| 실 윈도우 수 | scratch | 합성 사전학습 |
|-------------:|--------:|-------------:|
| 100 | 7.47 dB | **8.23 dB** |
| 300 | 7.96 dB | **8.71 dB** |
| 1000 | 8.78 dB | **9.09 dB** |
| 8000 | 9.65 dB | 9.58 dB |

- **실데이터 100개만으로 고전(6.70)·합성전용(4.09)을 넘는다.**
- 합성 사전학습은 저데이터 구간에서 이득 최대(+0.75 dB), 데이터 많아지면 수렴.
- 즉 합성의 올바른 쓰임새는 최종 모델이 아니라 **부트스트랩(사전학습 prior)**.

![data efficiency](assets/09_finetune_curve.png)

## 디노이징을 넘어 — 부정맥 분류

디노이저는 수단, 진단이 목적. MIT-BIH 박동 주석으로 각 심박을 AAMI 5클래스
(N/S/V/F/Q)로 분류. 3.6만 파라미터 1D-CNN, **환자 독립 분할**(레코드 분리)로 누수
차단 (`scripts/10_arrhythmia.py`).

- 미지 환자 박동 ~3.3만 개에서 전체 정확도 **81.8%**
- **PVC(V) recall 87%** — 임상적으로 중요한 이상 박동을 잘 잡음
- 희귀 클래스(S/F/Q)는 극심한 불균형(N≈3만 vs S≈260, F≈390)으로 저조 — 정직한 한계

![confusion matrix](assets/10_arrhythmia_cm.png)

## 디노이징이 진단을 *해칠* 때 (시스템 레벨 "깨끗하지만 틀림")

당연해 보이는 파이프라인은 디노이징 → 분류다. 노이즈로 잃은 정확도를 디노이징이
되살리는가? **아니다 — 오히려 나빠진다** (`scripts/12_denoise_then_classify.py`).

| 입력 SNR | 정확도 noisy → denoised | PVC recall noisy → denoised |
|---------:|:------------------------|:----------------------------|
| 9 dB | 79.0% → **74.3%** | 74.2% → **41.7%** |
| 6 dB | 75.0% → 72.1% | 66.8% → 36.3% |
| clean 상한 | 81.8% | 86.8% |

피해는 하필 가장 중요한 곳에서 크다 — **PVC recall이 반토막**. 노이즈 아티팩트도
아니다: *깨끗한* PVC 박동을 디노이저에만 통과시켜도 PVC recall이 **86.8% → 58.0%**.

**왜:** SNR 최적화 디노이저는 *평균적·정상* ECG 형태를 학습해 모든 걸 그쪽으로
끌어당긴다 — PVC를 정의하는 비정상 형태를 감쇠시킨다. 신호를 더 깨끗하고 *더
정상처럼* 만드는데, 이상 박동 검출엔 정확히 반대다. (두 메커니즘이 겹친다: 디노이저가
병적 특징을 뭉개고, 그 출력은 clean 학습 분류기 입장에서 OOD다.)

![pvc morphology](assets/12_pvc_morphology.png)

교훈: **부분 최적화는 조립되지 않는다.** 디노이저를 fidelity로, 분류기를 clean
정확도로 각각 최적화한 뒤 이으면 전체가 나빠진다. 파이프라인은 과제 인식 설계(공동
학습, 또는 SNR이 아니라 진단 특징을 보존하는 목적함수)가 필요하다. 이 프로젝트 주제의
시스템 레벨 결정판: *지표 최적 부품이 시스템에는 목적-오답일 수 있다.*

![denoise then classify](assets/12_denoise_then_classify.png)

### 해결 — 과제 인식 / 공동 학습

문제가 목적 불일치라면, 목적을 정렬하면 된다. 세 처방을 검증
(`scripts/13_joint_training.py`), **균형 정확도**(macro-recall)와 **PVC recall**로 평가:

| 방법 (입력 9 dB) | 균형 정확도 | PVC recall |
|------------------|-----------:|-----------:|
| naive (noisy→clean 분류기) | 41.1% | 73.0% |
| broken (디노이즈→clean 분류기) | 35.7% | 42.4% |
| noise-aware 분류기 | 35.7% | 84.6% |
| **joint (end-to-end)** | **41.2%** | **84.8%** |
| clean 상한 | 45.2% | 86.8% |

- **end-to-end 공동학습만이 두 축 모두 clean에 근접** — PVC recall 42→85%, 균형
  정확도 희생 없이.
- noise-aware는 PVC는 살리나 과편향(균형 정확도는 broken 수준). broken은 둘 다 최악.
- 같은 아키텍처·데이터인데 *목적 정렬*만 바꿔 42%가 85%로.

![fixing the coupling](assets/13_joint_training.png)

원리: *파이프라인 성능은 부품 품질이 아니라 부품과 시스템 목적의 정렬에서 나온다.*

## 도메인 시프트 — ML의 함정

원분포(정상 형태)로만 학습한 ML을, **형태가 다른 분포**(T파 역전·QRS 확대·이소성
박동)에서 테스트하면 결과가 뒤집힌다.

| input SNR | classical | ML (원분포 학습) |
|----------:|----------:|-----------------:|
|  +5.2 dB  | **12.1 dB** | 2.4 dB         |
|  −0.8 dB  | **10.2 dB** | 2.5 dB         |
|  −4.3 dB  | **8.3 dB**  | 2.9 dB         |

- 인분포에서 15 dB 압승하던 ML이 **OOD에서는 고전에 크게 진다.**
- 원인: ML이 학습한 형태 사전지식을 강요해 **역전 T파를 정상 T파로 왜곡**한다.
  겉보기엔 깨끗하지만 임상적으로 중요한 형태가 틀린다 — 의료 응용의 안전 이슈.
- 고전 필터는 분포 가정이 없어 **강건**하다.

이것이 이 프로젝트의 핵심 교훈이다: *ML의 우위는 학습분포 안에서만 유효하며, 일반화
비용(generalization tax)을 반드시 함께 봐야 한다.*

![domain shift](assets/04_domain_shift.png)

### 해결 — 혼합 분포 학습

두 형태를 **함께 학습**시키면 일반화가 회복된다.

| 모델 | 원분포 테스트 | 변형분포 테스트 |
|------|-------------:|---------------:|
| 원분포만 학습 | **17.7 dB** | 2.4 dB |
| 혼합 학습     | 13.7 dB     | **12.3 dB** |

- 혼합 학습은 인분포에서 4 dB를 양보하지만, **OOD 성능을 2.4 → 12.3 dB로 되산다.**
- 데이터 다양성이 일반화 비용을 지불하는 가장 직접적인 수단임을 정량적으로 보여준다.
- 실전 함의: *모델의 강건함은 아키텍처보다 학습분포가 얼마나 실제를 덮느냐에 달렸다.*

## 틀릴 때를 아는가 — 불확실도 안전 게이트

위의 도메인 시프트 실패가 위험했던 이유는 **조용했기** 때문이다. 모델이 자기 OOD
입력을 스스로 플래그할 수 있을까? 두 방법을 같은 테스트(in-dist vs 변형)로 비교.

| 방법 | OOD AUROC | 탐지율@5%오탐 |
|------|----------:|-------------:|
| MC-dropout | 0.58 | 9% |
| **딥 앙상블 (K=4)** | **0.90** | **50%** |

- **MC-dropout은 거의 무작위** — dropout 불확실도는 국소적으로 그럴듯한 미묘한 OOD에 둔감.
- **딥 앙상블의 불일치는 잡아낸다**(AUROC 0.90). 다른 시드로 학습한 멤버들이 인분포에선
  일치하나 낯선 형태에서 갈라진다.
- 실전: 디노이저에 앙상블 게이트를 붙여 불일치 큰 윈도우를 고전 DSP나 사람 검토로 라우팅.
  "깨끗하지만 틀린" 출력을 잡을 수 있게 된다.

![uncertainty](assets/07_uncertainty.png)

**게이트가 실제 sim-to-real 실패를 잡는가?** 잡는다. 합성 학습 앙상블이 **실**
MIT-BIH 데이터를 OOD로 **AUROC 0.96**에 감지 — 실데이터 불일치가 in-dist의 3.0배,
5% 오탐 게이트로 실입력의 **80.7%**를 잡는다 (`scripts/11_realdata_gate.py`). 즉
합성 모델이 실데이터에서 조용히 실패한 것을 배포 단계에서 **미리 잡아** 고전 폴백으로
돌릴 수 있다.

![real-data gate](assets/11_realdata_gate.png)

## 엣지 배포 타당성 (임베디드)

디노이저가 실시간으로 돌 수 있는가? (`scripts/06_edge_profile.py`, 단일 스레드)

| 항목 | 값 |
|------|----|
| 파라미터 | 56,289 |
| 메모리 (float32 / int8) | 220 KB / 55 KB |
| 연산량 | 55.9 kMAC/sample (≈20 MMAC/s) |
| 지연 (1024 윈도우, 1코어) | 2.4 ms |
| 실시간 계수(RTF) | 0.0009 (**1161× 실시간**) |

- 초당 약 20 MMAC 규모 → **int8 양자화 시 Cortex-M/저가 MPU급에서도 실시간이
  현실적.** 데이터가 발생하는 곳(웨어러블·베드사이드·기기 내부)에서 바로 추론 가능.
- DSP + ML + **임베디드 실시간**이 만나는 지점 — 이 프로젝트의 확장 방향.

## 빠른 시작
```bash
pip install -r requirements.txt

# 1) 고전 DSP 베이스라인 (즉시 실행)
python scripts/01_classical_denoise.py

# 2) ML 디노이저 학습 (CPU 몇 분)
python scripts/02_train_ml_denoise.py --n 4000 --epochs 30

# 3) 고전 vs ML 벤치마크
python scripts/03_benchmark.py

pytest -q   # 테스트
```
그림/표는 `outputs/`, 지표는 콘솔에 출력된다.
